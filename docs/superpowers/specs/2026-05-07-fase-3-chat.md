**Próxima fase:** F4 (Listagem + Download).

## 1. Objetivo

Entregar duas modalidades de chat com LLM sobre as notas do usuário:

- **Chat por nota** (`/documents/[id]`): thread **persistente única** por (user, doc), acoplada à página do documento. Responde perguntas sobre aquela invoice. Hidrata ao reabrir; botão `Limpar` apaga as mensagens server-side.
- **Chat global** (`/chat`): thread **persistente** com múltiplas sessões navegáveis em sidebar tipo ChatGPT. O modelo enxerga todas as notas do usuário (resumos no contexto inicial) e pode buscar texto completo de qualquer uma sob demanda via tool `get_full_document(id)`.

Ao fim da F3:

- Páginas `/documents/[id]` ganham o painel de chat persistente (slot já reservado em F2 como tab `Conversa` desabilitada).
- Rota nova `/chat` entra no ar, listando sessões na sidebar e o painel principal funcionando.
- Topbar (`V2Topbar` da F1) ativa o link `Chat` que estava `aria-disabled` desde F1.
- `ChatService` no Nest com OpenAI SDK direto, function calling, ownership check obrigatório dentro da tool, anti-injection com delimitadores XML.
- Streaming **opt-in** via env (`CHAT_STREAMING=false` por default; ambos caminhos implementados).
- Suite de testes cobre fluxo com e sem tool call, ownership, throttle, e duas jornadas E2E (chat de nota + chat global).

> **Alinhamento com o plano-mestre.** `ChatSession.documentId` é opcional: `NULL` para WORKSPACE, populado para DOCUMENT. Postgres trata múltiplos NULLs como distintos, então `@@unique([userId, documentId])` permite N sessões workspace por user e exige no máximo uma sessão por (user, doc). UX do chat-de-doc continua simples (sem switcher, sem histórico de N conversas — ver D4) — persistência server-side é ortogonal à UX. Esse alinhamento é o que viabiliza a F4 (download por doc com transcript do chat) sem hacks.

> **Histórico desta spec.** A versão 1 desta spec divergia do plano-mestre tornando o chat-de-doc stateless (D2/D4 originais). Durante o brainstorm da F4 (2026-05-09) percebemos que o requisito Invoice OCR ("download with appended extracted text and LLM interactions") força persistência do chat-de-doc. Como F3 ainda não havia sido implementada, corrigimos a spec antes do código sair. Decisões editadas: D2, D4, D6, D11, D20. Schema editado em §4. Endpoints editados em §5.4. DOCUMENT request shape e `useDocumentChat` editados.

| D1  | Streaming              | Opt-in via env `CHAT_STREAMING` (default `false`). Ambos caminhos no servidor; cliente solicita SSE com `Accept: text/event-stream`. Quando flag está `false`, header é ignorado e resposta é JSON.                                                                                                                                             |
| D2  | Modos de chat          | Dois modos discriminados pelo campo `ChatSession.documentId`: **WORKSPACE** (`documentId IS NULL`) e **DOCUMENT** (`documentId` populado). UX simples: sem switcher, sem `+ nova conversa`, sem histórico de N conversas — ao abrir o doc, hidrata a única sessão existente (cria-se na 1ª mensagem). | D7  | Provider LLM           | OpenAI SDK direto (`openai` npm), sem LangChain. Interface `LlmProvider` + `OpenaiLlmProvider` (default em prod) + `MockLlmProvider` (default em dev/test). | D13 | Janela de contexto     | Últimas `CHAT_MAX_HISTORY=20` mensagens enviadas ao modelo + system prompt. Mensagens `role=TOOL` contam na janela. WORKSPACE: lê do DB. DOCUMENT: client envia histórico já truncado em 20 mensagens.                                                                                                                                          |
| D14 | Mock provider          | `MockLlmProvider` retorna respostas determinísticas de teste. Dispara tool call quando user msg contém "texto completo" ou "valor total". Selecionado via `LLM_PROVIDER=mock` (default em `NODE_ENV=test`).                                                                                                                                     |
| D15 | UI base                | **Blazity shadcn-chatbot-kit** (MIT, registry shadcn-chatbot-kit.vercel.app) como base. Componentes instalados via `pnpm dlx shadcn@latest add`. Provider-agnostic — wiring com hooks próprios.                                                                                                                                                 |
| D16 | Tool iterations safety | Loop de tool calls limitado a `CHAT_MAX_TOOL_ITERATIONS=3`. Excedeu → erro 500 + log estruturado. Cada iteração consome 1 round-trip OpenAI.                                                                                                                                                                                                    |
| D17 | Tool messages na UI    | `GET /sessions/:id/messages` filtra `role≠TOOL` por padrão. Param `?includeTool=true` libera para auditoria (não consumido pelo cliente F3 — só pelo F4 download e debug manual).                                                                                                                                                               |
| D18 | Delete de sessões      | Backlog. F3 não entrega `DELETE /sessions/:id`. Cascade no User cumpre LGPD; gestão manual de cleanup do user vira UX da F-futura.                                                                                                                                                                                                              |
| D19 | Title da sessão        | Auto-gerado: `content` do primeiro user message truncado em 50 chars (com `…` se cortou). Atualizado uma vez, na 1ª mensagem persistida. Sem chamada LLM extra.                                                                                                                                                                                 |
| D20 | DOCUMENT request shape | Mesmo shape do WORKSPACE. `POST /chat/documents/:documentId/messages` com `{ content: string (1..8000) }`. Server hidrata (ou cria) a única sessão `(userId, documentId)`, persiste user + assistant + tool messages. Sem `history` no body — vem do DB.                                                                                        |

## 4. Schema Prisma (delta da F3)

Adicionar ao `apps/api/prisma/schema.prisma`:

```prisma
enum ChatRole {
  USER
  ASSISTANT
  TOOL
}

model ChatSession {
  id         String        @id @default(cuid())
  userId     String
  user       User          @relation(fields: [userId], references: [id], onDelete: Cascade)

  documentId String?
  document   Document?     @relation(fields: [documentId], references: [id], onDelete: Cascade)

  title      String?       // auto-gerado da 1ª user msg, truncado 50 chars (ver D19); null até a 1ª mensagem
  messages   ChatMessage[]

  createdAt  DateTime      @default(now())
  updatedAt  DateTime      @updatedAt

  @@unique([userId, documentId])  // NULL múltiplo OK (workspace), uma só por (user, doc)
  @@index([userId, updatedAt])
}

model ChatMessage {
  id          String      @id @default(cuid())
  sessionId   String
  session     ChatSession @relation(fields: [sessionId], references: [id], onDelete: Cascade)

  role        ChatRole
  content     String      @db.Text  // texto da mensagem; em role=TOOL guarda o output JSON serializado da tool

  toolCallId  String?     // OpenAI tool_call_id — preenchido em role=ASSISTANT (que disparou) e role=TOOL (que respondeu)
  toolName    String?     // nome da função ex.: 'get_full_document'
  toolArgs    Json?       // args que o assistant passou; auditoria

  createdAt   DateTime    @default(now())

  @@index([sessionId, createdAt])
}
```

Back-relations:

```prisma
model User {
  // ...campos da F1...
  documents     Document[]
  chatSessions  ChatSession[]
}

model Document {
  // ...campos da F2...
  chatSessions  ChatSession[]
}
```

**Migration:** `prisma migrate dev --name f3_chat`. Versionada no git.

**Notas do schema:**

- `Document` ganha back-relation com `ChatSession` (uma sessão por user, doc — ver D4). Cascade `onDelete` quando o doc é apagado.
- `onDelete: Cascade` cumpre LGPD em três níveis: User → Sessions, Document → Sessions (chat-de-doc), Session → Messages.
- `@@unique([userId, documentId])` discrimina os dois modos sem coluna `mode`. Postgres trata múltiplos NULLs como distintos: workspace permite N por user, document permite no máximo 1 por (user, doc).
- `ChatMessage.content` é `Text` (sem limite curto). Mensagens da role `TOOL` guardam `JSON.stringify(toolOutput)` — em F3 sempre `{ extractedText: ... }`, mas o campo é flexível para futuras tools.
- Índice `(userId, updatedAt)` em `ChatSession` cobre o list para sidebar do workspace (ordem desc) e o lookup do chat-de-doc.
- Índice `(sessionId, createdAt)` em `ChatMessage` cobre o hidrata-histórico (ordem asc).
- Não há constraint UNIQUE em `ChatMessage` — nada impede mensagens com mesmo `toolCallId` (são par assistant+tool).

## 5. Arquitetura e fluxo

### 5.1 Fluxo ponta a ponta — modo WORKSPACE (persistente)

```
[Browser]              [Next.js apps/web]              [Nest apps/api]                [OpenAI]   [Postgres]
   │                          │                              │                            │           │
   │── click "+ nova" ───────▶│ useWorkspaceChat.createNew    │                            │           │
   │                          │  POST /api/chat/sessions      │                            │           │
   │                          │   → apiFetch('/v1/chat/sessions', POST)                   │           │
   │                          │                              │ JwtAuthGuard → req.user      │           │
   │                          │                              │ Throttle@15/min chat         │           │
   │                          │                              │ ChatController.createSession:│           │
   │                          │                              │   prisma.chatSession.create  ─────────────▶│
   │                          │                              │ → 201 { id, createdAt }      │           │
   │                          │ router.push('/chat/' + id)    │                              │           │
   │                          │                              │                              │           │
   │── digita & Enter ───────▶│ useWorkspaceChat.send         │                              │           │
   │                          │  POST /api/chat/sessions/:id/messages                       │           │
   │                          │   Accept: text/event-stream (se CHAT_STREAMING=true)        │           │
   │                          │                              │ ChatController.sendWorkspaceMessage:    │
   │                          │                              │  ownership: chatSession.findFirst({ id, userId }) ──▶│
   │                          │                              │  load history (last N) ─────────────────────────────▶│
   │                          │                              │  load doc summaries (this user) ────────────────────▶│
   │                          │                              │  build system prompt (WORKSPACE variant)             │
   │                          │                              │  persist user msg ──────────────────────────────────▶│
   │                          │                              │  ┌── runConversation loop (max 3 tool iters) ──┐     │
   │                          │                              │  │  llm.complete({tools, messages}) ───▶│ resp │     │

   │                          │                              │  │    persist final ASSISTANT ─────────────────────▶│
   │                          │                              │  └────────────────────────────────────────────────┘ │
   │                          │                              │  update session.title (1ª msg) + updatedAt         │
   │                          │ ◀── 200 { content } ─────────│                              │           │
   │                          │     ou SSE { delta }*        │                              │           │
   │                          │ append to messages state     │                              │           │
   │                          │                              │                              │           │
   │── click sessão antiga ──▶│ useWorkspaceChat.select(id)   │                              │           │
   │                          │  GET /api/chat/sessions/:id/messages?includeTool=false      │           │
   │                          │   → apiFetch('/v1/chat/sessions/:id/messages')              │           │
   │                          │                              │ ownership + findMany(role≠TOOL)─▶│        │
   │                          │ ◀── 200 [...]                 │                              │           │
   │                          │ hydrate state                │                              │           │
```

\* Em modo stream, `204` headers seguidos de chunks SSE `data: { "delta": "..." }\n\n` até `data: [DONE]\n\n`.

### 5.2 Fluxo ponta a ponta — modo DOCUMENT (persistente)

```
[Browser /documents/[id]]          [Next.js apps/web]              [Nest apps/api]               [OpenAI]   [Postgres]
   │                                      │                              │                          │           │
   │── abre página ──────────────────────▶│ useDocumentChat (mount)       │                          │           │
   │                                      │  GET /api/chat/documents/:id/messages                    │           │
   │                                      │                              │ ownership doc + findFirst session(userId, documentId) ─▶│
   │                                      │                              │ findMany messages (role≠TOOL) ─────────▶│
   │                                      │ ◀── 200 [...messages]         │                              │           │
   │                                      │ hydrate state                │                              │           │
   │                                      │                              │                              │           │
   │── digita & Enter ───────────────────▶│ useDocumentChat.send          │                          │           │
   │                                      │  POST /api/chat/documents/:id/messages                   │           │
   │                                      │   body: { content }           │                          │           │
   │                                      │   → apiFetch('/v1/chat/documents/:id/messages')          │           │
   │                                      │                              │ JwtAuthGuard               │           │
   │                                      │                              │ Throttle@15/min chat       │           │
   │                                      │                              │ ChatController.sendDocumentMessage:    │
   │                                      │                              │  doc = doc.findFirst({ id, userId }) ─▶│
   │                                      │                              │  if !doc → 404; if status≠READY → 409 │
   │                                      │                              │  session = upsert by (userId, documentId) ─▶│
   │                                      │                              │  build system prompt (DOCUMENT, summary inline)
   │                                      │                              │  load history (last N from DB) ─▶│
   │                                      │                              │  persist user msg ────────────▶│
   │                                      │                              │  ┌── runConversation (com persist) ──┐│
   │                                      │                              │  │  llm.complete({tools, messages})  ││
   │                                      │                              │  │  if tool_calls → exec + persist ──▶││ getFullDocument(id, userId)
   │                                      │                              │  │  re-call até resposta final       ││
   │                                      │                              │  │  persist final assistant ────────▶││
   │                                      │                              │  └─────────────────────────────────┘ │
   │                                      │                              │  bump session.updatedAt ─▶│
   │                                      │ ◀── 200 { content } ─────────│                              │           │
   │                                      │     ou SSE { delta }*        │                              │           │
   │                                      │ append to messages state     │                              │           │
   │                                      │                              │                              │           │
   │── click "Limpar" ───────────────────▶│ DELETE /api/chat/documents/:id/messages                  │           │
   │                                      │                              │ ownership + deleteMany on session ─▶│
   │                                      │ ◀── 204                       │                              │           │
   │                                      │ setMessages([])              │                              │           │
```

**Diferenças vs WORKSPACE:**

- `documentId` populado em `ChatSession` (em vez de `NULL`); discriminador dos dois modos.
- Sessão é criada lazy na 1ª mensagem (upsert); GET inicial pode retornar `[]` se ainda não há sessão.
- System prompt builder usa `buildDocumentSystem(doc)` (resumo inline do doc específico), não `buildWorkspaceSystem(docs[])`.
- Não há sidebar nem `+ nova conversa`; UI permanece simples (D4) — switcher e múltiplas conversas continuam fora do escopo.
- `Limpar` agora é uma chamada de rede que apaga as mensagens da sessão (sessão sobrevive).

**Por que `chat` separa de `ocr` e `documents`:** confirmado em CLAUDE.md e na spec Invoice OCR. Responsabilidades diferentes (OCR é one-shot pesado; chat é multi-turn leve com tool dispatching), evolução independente. F3 importa `prisma.document` via `documents.service` (read-only) e nunca toca `ocr`.

**`LlmProvider` injetado por token** (`LLM_PROVIDER`), não por classe — permite swap por mock em tests sem mexer em consumidor.

**`tools-registry`:** mapa `Record<string, { schema: ToolSchema, handler: (args, ctx: { userId }) => Promise<unknown> }>`. F3 registra só `get_full_document`. Adicionar tool no futuro = adicionar entrada no registry.

### 5.4 Endpoints

Ownership obrigatório. `Accept: text/event-stream` ativa SSE quando `CHAT_STREAMING=true` — emite `data: { "delta": "..." }\n\n` chunks; encerra com `data: [DONE]\n\n`. Tool round-trips são server-side em ambos modos. |
| GET    | `/api/v1/chat/documents/:documentId/messages?includeTool=false` | Bearer     | 60/min default | `200 [{ id, role, content, createdAt }, ...]`. Ownership do doc. Carrega a sessão única `(userId, documentId)`; se ainda não existe, retorna `[]`. `role≠TOOL` por padrão. Order `createdAt asc`.                                                                                        |
| POST   | `/api/v1/chat/documents/:documentId/messages`                   | Bearer     | 15/min `chat`  | `200 { content }`. Body `{ content: string (1..8000) }`. Ownership obrigatório; `409 { code: 'document_not_ready' }` se `doc.status ≠ READY`. Upsert da sessão `(userId, documentId)`. Persiste user + assistant + tool. SSE quando `CHAT_STREAMING=true`.                               |
| DELETE | `/api/v1/chat/documents/:documentId/messages`                   | Bearer     | 15/min `chat`  | `204`. Ownership do doc. Apaga todas as mensagens da sessão `(userId, documentId)` se existir; sessão sobrevive. No-op se sessão não existe.                                                                                                                                             |

**Códigos de erro padronizados:**

- `401` — Bearer ausente/inválido (handler do `JwtAuthGuard`).
- `404` — sessão/doc inexistente OU não pertencente ao user (mesma resposta para evitar enumeration).
- `429` — throttle excedido.
- `503` — provider LLM fora do ar (timeout, 5xx repetidos). Body `{ code: 'provider_error' }`.
- `500` — loop de tool iterations excedido (`CHAT_MAX_TOOL_ITERATIONS`). Logado como `chat.tool_loop_exceeded` com `sessionId`.

### 5.5 Pipeline interno do `ChatService.runConversation`

Função reutilizada por ambos os modos. Recebe um "context bundle" e retorna a resposta final do assistant.

```ts
type RunContext = {
  userId: string;
  systemPrompt: string;          // já montado pelo builder específico do modo
  messages: ChatMessage[];        // últimas N msgs (já truncadas) carregadas do DB
  persist: (msg: PendingMessage) => Promise<ChatMessage>; // sempre persiste — closure ligada à sessão
  onAssistantDelta?: (chunk: string) => void;              // só quando streaming
};

async runConversation(ctx: RunContext): Promise<{ content: string }> {
  let iter = 0;
  let conversation = [
    { role: 'system', content: ctx.systemPrompt },
    ...ctx.messages.map(toOpenAiMessage),
  ];

  while (iter < this.config.maxToolIterations) {
    const resp = await this.llm.complete({
      model: this.config.chatModel,
      messages: conversation,
      tools: this.toolsRegistry.getOpenAiSchemas(),
      stream: !!ctx.onAssistantDelta,
    });

    // Streaming: agrega chunks, mas NÃO emite no cliente até saber que é msg final (não tool_call)
    const message = ctx.onAssistantDelta
      ? await this.collectStream(resp, ctx.onAssistantDelta)
      : resp.choices[0].message;

    if (message.tool_calls?.length) {
      // Persiste o assistant que disparou tool(s), uma row por iter
      const persistedAsst = await ctx.persist({
        role: 'ASSISTANT',
        content: message.content ?? '',  // pode ser '' quando só dispara tools
        toolCallId: message.tool_calls[0].id, // primeira (F3 só usa 1 tool, OpenAI raramente paraleliza)
        toolName: message.tool_calls[0].function.name,
        toolArgs: JSON.parse(message.tool_calls[0].function.arguments),
      });

      conversation.push(message);

      // Executa cada tool com ownership check via context
      for (const call of message.tool_calls) {
        const handler = this.toolsRegistry.getHandler(call.function.name);
        if (!handler) throw new Error('unknown_tool');
        const args = JSON.parse(call.function.arguments);
        const output = await handler(args, { userId: ctx.userId });
        const outputJson = JSON.stringify(output);

        await ctx.persist({
          role: 'TOOL',
          content: outputJson,
          toolCallId: call.id,
          toolName: call.function.name,
        });

        conversation.push({
          role: 'tool',
          tool_call_id: call.id,
          content: outputJson,
        });
      }

      iter++;
      continue;
    }

    // Resposta final
    await ctx.persist({ role: 'ASSISTANT', content: message.content ?? '' });
    return { content: message.content ?? '' };
  }

  this.logger.error({ event: 'chat.tool_loop_exceeded', userId: ctx.userId, iter });
  throw new InternalServerErrorException('tool_loop_exceeded');
}
```

**Notas de design:**

- `persist` é **sempre** definido — closure que chama `prisma.chatMessage.create` com `sessionId` capturado pelo controller. Os dois modos (WORKSPACE e DOCUMENT) compartilham o mesmo loop e o mesmo comportamento de persistência; só diferem no `sessionId` da closure (e no `systemPrompt` montado).
- `onAssistantDelta` ativa o caminho streaming. Em iterações que disparam tools, o callback **não** é chamado (chunks são coletados internamente, mas não vazam para o cliente — assim o usuário não vê o conteúdo "parcial" de uma chamada de tool).
- Ownership check vive no handler da tool (`get-full-document.tool.ts`), recebendo `userId` via `ctx`. Dupla proteção: o modelo não pode "burlar" filtrando por ID alheio porque a query Prisma sempre carrega o `userId` real do request.
- Erros transientes da OpenAI (rate-limit, 5xx, timeout) borbulham para o controller; um interceptor mapeia para `503`. Loop `iter` não conta tentativas falhas — é só pra tool calls bem-sucedidos.

```ts
@Injectable()
export class ChatService {
  constructor(
    private readonly prisma: PrismaService,
    @Inject(LLM_PROVIDER) private readonly llm: LlmProvider,
    private readonly toolsRegistry: ToolsRegistry,
    private readonly config: ConfigService,
    private readonly logger: Logger,
  ) {}

  get streamingEnabled() {
    return this.config.get('CHAT_STREAMING') === 'true';
  }

  async createSession(userId: string) {
    return this.prisma.chatSession.create({
      data: { userId },
      select: { id: true, createdAt: true },
    });
  }

  async listSessions(userId: string, limit: number) {
    return this.prisma.chatSession.findMany({
      where: { userId },
      select: { id: true, title: true, createdAt: true, updatedAt: true },
      orderBy: { updatedAt: 'desc' },
      take: Math.min(limit, 100),
    });
  }

  async listMessages(userId: string, sessionId: string, includeTool: boolean) {
    // ownership check via chatSession.findFirst({ id, userId })
    const session = await this.prisma.chatSession.findFirst({
      where: { id: sessionId, userId },
      select: { id: true },
    });
    if (!session) throw new NotFoundException();

    return this.prisma.chatMessage.findMany({
      where: {
        sessionId,
        ...(includeTool ? {} : { role: { not: 'TOOL' } }),
      },
      orderBy: { createdAt: 'asc' },
      select: { id: true, role: true, content: true, createdAt: true },
    });
  }

  async sendWorkspaceMessage(userId: string, sessionId: string, content: string) {
    const session = await this.prisma.chatSession.findFirst({
      where: { id: sessionId, userId },
    });
    if (!session) throw new NotFoundException();

    // Persiste user msg primeiro — sobrevive a falha de LLM
    await this.prisma.chatMessage.create({
      data: { sessionId, role: 'USER', content },
    });

    // Carrega histórico (últimas N) e summaries dos docs do user
    const history = await this.loadHistory(sessionId);
    const docs = await this.prisma.document.findMany({
      where: { userId, status: 'READY' },
      select: { id: true, filename: true, summary: true },
      orderBy: { updatedAt: 'desc' },
      take: 50, // limite prático
    });

    const systemPrompt = buildWorkspaceSystem(docs);

    const result = await this.runConversation({
      userId,
      systemPrompt,
      messages: history,
      persist: (msg) => this.prisma.chatMessage.create({ data: { sessionId, ...msg } }),
    });

    // Title da 1ª msg + bump updatedAt
    if (!session.title) {
      await this.prisma.chatSession.update({
        where: { id: sessionId },
        data: { title: titleFromContent(content), updatedAt: new Date() },
      });
    } else {
      await this.prisma.chatSession.update({
        where: { id: sessionId },
        data: { updatedAt: new Date() },
      });
    }

    return { content: result.content };
  }

  async sendDocumentMessage(userId: string, documentId: string, content: string) {
    const doc = await this.prisma.document.findFirst({
      where: { id: documentId, userId },
      select: { id: true, filename: true, summary: true, status: true },
    });
    if (!doc) throw new NotFoundException();
    if (doc.status !== 'READY') {
      throw new ConflictException({ code: 'document_not_ready' });
    }

    // Upsert da sessão única (userId, documentId)
    const session = await this.prisma.chatSession.upsert({
      where: { userId_documentId: { userId, documentId } },
      create: { userId, documentId },
      update: {},
    });

    // Persiste user msg primeiro — sobrevive a falha de LLM
    await this.prisma.chatMessage.create({
      data: { sessionId: session.id, role: 'USER', content },
    });

    const history = await this.loadHistory(session.id);
    const systemPrompt = buildDocumentSystem(doc);

    const result = await this.runConversation({
      userId,
      systemPrompt,
      messages: history,
      persist: (msg) => this.prisma.chatMessage.create({ data: { sessionId: session.id, ...msg } }),
    });

    await this.prisma.chatSession.update({
      where: { id: session.id },
      data: { updatedAt: new Date() },
    });

    return { content: result.content };
  }

  async listDocumentMessages(userId: string, documentId: string, includeTool: boolean) {
    const doc = await this.prisma.document.findFirst({
      where: { id: documentId, userId },
      select: { id: true },
    });
    if (!doc) throw new NotFoundException();

    const session = await this.prisma.chatSession.findFirst({
      where: { userId, documentId },
      select: { id: true },
    });
    if (!session) return [];

    return this.prisma.chatMessage.findMany({
      where: {
        sessionId: session.id,
        ...(includeTool ? {} : { role: { not: 'TOOL' } }),
      },
      orderBy: { createdAt: 'asc' },
      select: { id: true, role: true, content: true, createdAt: true },
    });
  }

  async clearDocumentMessages(userId: string, documentId: string) {
    const doc = await this.prisma.document.findFirst({
      where: { id: documentId, userId },
      select: { id: true },
    });
    if (!doc) throw new NotFoundException();

    const session = await this.prisma.chatSession.findFirst({
      where: { userId, documentId },
      select: { id: true },
    });
    if (!session) return; // no-op

    await this.prisma.chatMessage.deleteMany({ where: { sessionId: session.id } });
  }

  // Variantes streaming (sendWorkspaceMessageStream / sendDocumentMessageStream) reutilizam runConversation
  // passando onAssistantDelta=(chunk) => res.write(`data: ${JSON.stringify({delta: chunk})}\n\n`)
  // e finalizam com res.write('data: [DONE]\n\n'); res.end();

  private async loadHistory(sessionId: string) {
    const max = this.config.get('CHAT_MAX_HISTORY');
    const all = await this.prisma.chatMessage.findMany({
      where: { sessionId }, // inclui TOOL msgs aqui — vão para o modelo
      orderBy: { createdAt: 'desc' },
      take: max,
    });
    return all.reverse();
  }
}
```

### 6.8 Tool `get_full_document`

```ts
// chat/tools/get-full-document.tool.ts
import { z } from 'zod';

const argsSchema = z.object({
  documentId: z.string().min(1),
});

@Injectable()
export class GetFullDocumentTool {
  static readonly name = 'get_full_document';
  static readonly schema = {
    type: 'function' as const,
    function: {
      name: 'get_full_document',
      description: 'Retorna o texto completo extraído de um documento do usuário.',
      parameters: {
        type: 'object',
        properties: {
          documentId: { type: 'string', description: 'ID do documento (cuid)' },
        },
        required: ['documentId'],
      },
    },
  };

  constructor(private readonly prisma: PrismaService) {}

  async execute(
    rawArgs: unknown,
    ctx: { userId: string },
  ): Promise<{ extractedText: string } | { error: string }> {
    const parsed = argsSchema.safeParse(rawArgs);
    if (!parsed.success) return { error: 'invalid_arguments' };

    // Ownership check obrigatório — userId vem do request, não do modelo
    const doc = await this.prisma.document.findFirst({
      where: { id: parsed.data.documentId, userId: ctx.userId },
      select: { extractedText: true, status: true },
    });

    if (!doc) return { error: 'not_found' };
    if (doc.status !== 'READY') return { error: 'not_ready' };
    if (!doc.extractedText) return { error: 'no_text' };

    return { extractedText: doc.extractedText };
  }
}
```

```ts
// chat/tools/tools-registry.ts
@Injectable()
export class ToolsRegistry {
  constructor(private readonly getFullDocument: GetFullDocumentTool) {}

  getOpenAiSchemas(): ChatCompletionTool[] {
    return [GetFullDocumentTool.schema];
  }

  getHandler(name: string) {
    if (name === GetFullDocumentTool.name) {
      return (args: unknown, ctx: { userId: string }) => this.getFullDocument.execute(args, ctx);
    }
    return null;
  }
}
```

**Decisão de design:** o tool **não lança** em erros previsíveis (`not_found`, `not_ready`) — retorna `{ error: '<code>' }` para o modelo. Isso permite ao modelo informar o usuário ("o documento que você pediu não existe") em vez de quebrar a conversa toda. Erros não-previsíveis (DB down) borbulham e viram 500.

**Logado:**

- `event` (ex.: `chat.message_sent`, `chat.tool_called`, `chat.tool_loop_exceeded`, `chat.session_created`).
- `userId`, `sessionId` (se WORKSPACE), `documentId` (se DOCUMENT), `latencyMs`, `tokensIn`, `tokensOut` (se OpenAI retornar).
- `toolName` quando aplicável.

**Nunca logado:**

- `content` de qualquer mensagem (user, assistant, tool output).
- `extractedText` ou `summary` de documento.
- `Authorization` header (interceptor F1 já mascara).

`PrismaService` já tem logger com sanitization de F2 (não loga `extractedText` em queries lentas) — F3 não precisa adicionar.

```ts
// components/features/chat/use-workspace-chat.ts
'use client';

import { useCallback, useEffect, useState } from 'react';
import { useRouter } from 'next/navigation';

export function useWorkspaceChat(activeSessionId?: string) {
  const router = useRouter();
  const [sessions, setSessions] = useState<Session[]>([]);
  const [messages, setMessages] = useState<Message[]>([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  // Load sessions on mount
  useEffect(() => {
    fetch('/api/chat/sessions')
      .then((r) => r.json())
      .then(setSessions);
  }, []);

  // Load messages when activeSessionId changes
  useEffect(() => {
    if (!activeSessionId) {
      setMessages([]);
      return;
    }
    fetch(`/api/chat/sessions/${activeSessionId}/messages`)
      .then((r) => r.json())
      .then(setMessages);
  }, [activeSessionId]);

  const createSession = useCallback(async () => {
    const res = await fetch('/api/chat/sessions', { method: 'POST' });
    const { id } = await res.json();
    router.push(`/chat/${id}`);
  }, [router]);

  const send = useCallback(
    async (content: string) => {
      if (!activeSessionId) return;
      setError(null);
      setLoading(true);

      const userMsg: Message = { id: `tmp-${Date.now()}`, role: 'USER', content };
      setMessages((prev) => [...prev, userMsg]);

      try {
        const res = await fetch(`/api/chat/sessions/${activeSessionId}/messages`, {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({ content }),
        });
        if (!res.ok) throw new Error(`http_${res.status}`);

        if (res.headers.get('content-type')?.includes('text/event-stream')) {
          // streaming path: leitor SSE acumula chunks num assistant msg
          const reader = res.body!.getReader();
          let acc = '';
          const tmpId = `asst-${Date.now()}`;
          setMessages((prev) => [...prev, { id: tmpId, role: 'ASSISTANT', content: '' }]);
          while (true) {
            const { value, done } = await reader.read();
            if (done) break;
            const chunk = parseSseChunk(value);
            if (chunk?.delta) {
              acc += chunk.delta;
              setMessages((prev) => prev.map((m) => (m.id === tmpId ? { ...m, content: acc } : m)));
            }
          }
        } else {
          const { content: asst } = await res.json();
          setMessages((prev) => [
            ...prev,
            { id: `asst-${Date.now()}`, role: 'ASSISTANT', content: asst },
          ]);
        }

        // Refresh sessions list (title pode ter sido auto-gerado)
        fetch('/api/chat/sessions')
          .then((r) => r.json())
          .then(setSessions);
      } catch (e: any) {
        setError(e?.message ?? 'unknown');
      } finally {
        setLoading(false);
      }
    },
    [activeSessionId],
  );

  return { sessions, messages, loading, error, createSession, send };
}
```

```ts
// components/features/chat/use-document-chat.ts
'use client';

import { useCallback, useEffect, useState } from 'react';

export function useDocumentChat(documentId: string) {
  const [messages, setMessages] = useState<Message[]>([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  // Hidrata mensagens da única sessão (userId, documentId) ao montar
  useEffect(() => {
    let alive = true;
    fetch(`/api/chat/documents/${documentId}/messages`)
      .then((r) => (r.ok ? (r.json() as Promise<Message[]>) : []))
      .then((rows) => {
        if (alive) setMessages(rows);
      })
      .catch(() => undefined);
    return () => {
      alive = false;
    };
  }, [documentId]);

  const send = useCallback(
    async (content: string) => {
      setError(null);
      setLoading(true);

      const userMsg: Message = { id: `u-${Date.now()}`, role: 'USER', content };
      setMessages((prev) => [...prev, userMsg]);

      try {
        const res = await fetch(`/api/chat/documents/${documentId}/messages`, {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({ content }),
        });
        if (!res.ok) throw new Error(`http_${res.status}`);

        if (res.headers.get('content-type')?.includes('text/event-stream')) {
          const reader = res.body!.getReader();
          let acc = '';
          const tmpId = `a-${Date.now()}`;
          setMessages((prev) => [...prev, { id: tmpId, role: 'ASSISTANT', content: '' }]);
          while (true) {
            const { value, done } = await reader.read();
            if (done) break;
            const chunk = parseSseChunk(value);
            if (chunk?.delta) {
              acc += chunk.delta;
              setMessages((prev) => prev.map((m) => (m.id === tmpId ? { ...m, content: acc } : m)));
            }
          }
        } else {
          const { content: asst } = await res.json();
          setMessages((prev) => [
            ...prev,
            { id: `a-${Date.now()}`, role: 'ASSISTANT', content: asst },
          ]);
        }
      } catch (e: any) {
        setError(e?.message ?? 'unknown');
      } finally {
        setLoading(false);
      }
    },
    [documentId],
  );

  const clear = useCallback(async () => {
    await fetch(`/api/chat/documents/${documentId}/messages`, { method: 'DELETE' });
    setMessages([]);
  }, [documentId]);

  return { messages, loading, error, send, clear };
}
```

## 9. Testes

### 9.1 Unit — Nest

- `ChatService.runConversation` (mock LLM):
  - 1 turn sem tool → persiste user + assistant; retorna content.
  - 1 turn com tool → persiste user + assistant(toolCallId) + tool + assistant final; retorna content.
  - 2 tool iterations → persiste 4 msgs após user + final.
  - 3+ iterations → erro `tool_loop_exceeded`.
- `ChatService.sendWorkspaceMessage`:
  - Ownership: 404 quando session.userId ≠ req.user.id.
  - Title auto-gerado na 1ª msg (truncado).
  - `updatedAt` bump.
- `ChatService.sendDocumentMessage`:
  - Ownership do doc (404).
  - 409 quando doc.status ≠ READY.
  - 1ª mensagem: cria sessão; 2ª mensagem reusa a mesma sessão (`@@unique([userId, documentId])`).
  - Persiste user + assistant no DB; histórico carregado do DB no turn seguinte.
- `ChatService.listDocumentMessages` / `clearDocumentMessages`:
  - Ownership do doc (404).
  - listDocument retorna `[]` quando sessão ainda não existe.
  - clear é no-op quando sessão não existe; apaga todas as mensagens quando existe; sessão sobrevive.
- `GetFullDocumentTool.execute`:
  - Ownership: doc do user → ok.
  - Doc de outro user → `{ error: 'not_found' }`.
  - Doc do user mas `status≠READY` → `{ error: 'not_ready' }`.
  - args inválidos → `{ error: 'invalid_arguments' }`.
- `OpenaiLlmProvider`:
  - Shape do request a `chat.completions.create` (model, messages, tools, tool_choice).
- `buildWorkspaceSystem` / `buildDocumentSystem`:
  - Inclui RULES, delimitadores XML, lista correta de docs.
  - Lista vazia (workspace) gera variante "sem docs".

### 9.2 Integração — Nest

- `POST /v1/chat/sessions` — 401 sem token, 201 com token, throttle no 16º request.
- `POST /v1/chat/sessions/:id/messages` (mock LLM):
  - 404 quando id inexistente.
  - 404 quando id de outro user.
  - Fluxo feliz sem tool: 200 + `{ content }`.
  - Fluxo com tool: 200 + `{ content }`, e DB tem 4 msgs (user, assistant-tool, tool, assistant final).
  - 429 throttle.
- `GET /v1/chat/documents/:id/messages` (mock LLM):
  - 401 sem token.
  - 404 doc de outro user.
  - Doc do user, sem sessão ainda → `200 []`.
  - Doc do user, sessão com 3 msgs → `200 [3 itens]` ordem `createdAt asc`, sem TOOL.
- `POST /v1/chat/documents/:id/messages` (mock LLM):
  - 404 doc de outro user.
  - 409 doc QUEUED/OCR_RUNNING.
  - 1ª chamada: cria sessão + persiste user + assistant.

  - 404 doc de outro user.
  - Sem sessão → 204 (no-op).

## 14. Emendas F2.5 (2026-05-09)

A F2.5 introduz o campo `summary.narrative` (string em linguagem natural gerada pelo extractor) e altera o schema `core` para campos universais. Os system prompt builders do F3 devem usar o `narrative` como contexto principal do documento, com fallback para o `core` serializado.

### 14.1 `buildWorkspaceSystem`

```ts
// Antes:
`${JSON.stringify(d.summary?.core ?? {})}`
// Depois:
`${d.summary?.narrative ?? JSON.stringify(d.summary?.core ?? {})}`;
```

### 14.2 `buildDocumentSystem`

```ts
// Antes:
`<document id="${doc.id}">
  <filename>${doc.filename}</filename>
  <summary>${JSON.stringify(doc.summary)}</summary>
</document>`
// Depois:
`<document id="${doc.id}">
  <filename>${doc.filename}</filename>
  <narrative>${doc.summary?.narrative ?? ''}</narrative>
  <summary>${JSON.stringify(doc.summary?.core ?? {})}</summary>
</document>`;
```

O `<narrative>` dá ao modelo contexto denso em linguagem natural. O `<summary>` mantém os campos estruturados para perguntas precisas (ex.: "qual o número da nota?"). Ambos são delimitados por tags XML — anti-injection mantido.

### 14.3 Campos `core` universais

O shape de `doc.summary?.core` muda do schema BR para o schema universal (F2.5 §5). O `buildWorkspaceSystem` e `buildDocumentSystem` não precisam de ajuste além do 14.1/14.2 acima — serializam `core` como JSON independente do shape.
