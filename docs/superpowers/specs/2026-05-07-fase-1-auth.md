# Fase 1 — Auth (Spec)

**Data:** 2026-05-07
**Fase anterior:** F0 (monorepo bootstrap, plano `2026-05-07-monorepo-bootstrap.md`) e F0.5 (skeleton, spec `2026-05-07-fase-0.5-skeleton.md` — pré-requisito).
**Próxima fase:** F2 (OCR — upload, vision, ver texto).

---

## 1. Objetivo

Entregar autenticação OAuth ponta-a-ponta com Google e GitHub, sessão JWT compartilhada entre `apps/web` (NextAuth) e `apps/api` (NestJS), persistência mínima de usuários no Postgres com RBAC `USER|ADMIN`, e o esqueleto autenticado da aplicação (página `/login` finalizada conforme design + topbar autenticada + página home minimal).

Ao fim da F1:

- Usuário consegue logar com Google ou GitHub e ser redirecionado para `/`.
- O Nest reconhece o usuário em qualquer endpoint protegido via `Authorization: Bearer <JWT>`.
- Logout funciona pelo dropdown da topbar e retorna para `/login`.
- E-mails listados em `ADMIN_EMAILS` são promovidos automaticamente a `ADMIN` no signIn.
- Suite de testes (unit + E2E) cobre os fluxos críticos de auth.

## 3. Decisões fechadas

| #   | Decisão                     | Escolha                                                                          |
| --- | --------------------------- | -------------------------------------------------------------------------------- |
| D1  | Localização da sessão       | NextAuth (`apps/web`), modo JWT (`session.strategy='jwt'`)                       |
| D2  | Transporte do token Web→API | `Authorization: Bearer <JWT>` em fetch server-side do Next                       |
| D3  | Persistência de sessão      | Sem Prisma adapter; só model `User`. Sem `Account`/`Session`/`VerificationToken` |
| D4  | Validação no Nest           | Decode com `NEXTAUTH_SECRET` compartilhado, busca `User` no DB pelo `sub`        |
| D5  | Bootstrap de ADMIN          | Lista de e-mails via env `ADMIN_EMAILS` (CSV); promovidos no `signIn` callback   |
| D6  | Redirect pós-login          | `/` (home autenticada minimal nesta fase; F2 transforma em Upload)               |
| D7  | Mock OAuth para E2E         | `CredentialsProvider` extra, registrado **apenas** quando `NODE_ENV === 'test'`  |
| D8  | Expiração da sessão         | `maxAge` 7 dias, `updateAge` 24h (sliding refresh)                               |
| D9  | Rate limit `/api/auth/*`    | 5 req/min por IP, in-memory na F1 (Upstash/Redis no backlog)                     |
| D10 | Provider de origem          | Não persistido; dedup por e-mail no `signIn` callback                            |

## 5. Fluxo de autenticação

```
[Browser]                [Next.js apps/web]                 [Nest apps/api]              [Postgres]
    │                            │                                 │                          │
    │── click "Google" ─────────▶│ /api/auth/signin/google         │                          │
    │                            │   ↓                              │                          │
    │                            │   redirect 302 → Google OAuth   │                          │
    │── consente ────────────────────────────────────────────▶ Google                          │
    │◀── 302 /api/auth/callback/google?code=... ──────────────────                              │
    │                            │ NextAuth troca code → tokens    │                          │
    │                            │ ↓ signIn callback                │                          │
    │                            │   prisma.user.upsert({ email })─────────────────────────▶  │
    │                            │   role = email∈ADMIN_EMAILS ? ADMIN : USER                  │
    │                            │ ↓ jwt callback (1ª vez)         │                          │
    │                            │   token.sub = user.id           │                          │
    │                            │   token.role = role             │                          │
    │                            │ ↓ assina JWT (HS256, NEXTAUTH_SECRET)                       │
    │◀── set-cookie next-auth.session-token=<JWT> ─ 302 /          │                          │
    │                            │                                 │                          │
    │── GET / ──────────────────▶│ middleware: cookie ok → next()  │                          │
    │                            │ RSC fetch /api/v1/me            │                          │
    │                            │   getToken({ req, secret }) → JWT                          │
    │                            │   fetch(API_URL, {              │                          │
    │                            │     headers: { Authorization: `Bearer ${jwt}` }            │
    │                            │   }) ───────────────────────────▶│                          │
    │                            │                                 │ JwtAuthGuard:            │
    │                            │                                 │  decode + verify HS256   │
    │                            │                                 │  → payload.sub           │
    │                            │                                 │  prisma.user.findUnique ─▶│
    │                            │                                 │  → req.user              │
    │                            │◀── { id, email, name, role } ───│                          │
    │◀── HTML "Bem-vindo, ..." ──│                                 │                          │
```

Notas:

- **Cookie nunca sai do browser pra api.** O Next lê o cookie server-side com `getToken()` (de `next-auth/jwt`) e injeta o JWT como `Authorization: Bearer` no fetch interno pra api. Cliente JS nunca toca o token cru.
- **CORS na api é simples**: `credentials: false`, `origin: ALLOWED_ORIGINS.split(',')`. Sem complicação de cross-origin cookie.

### 6.3 `lib/auth.ts` (esboço — conteúdo final na implementação)

```ts
import { type NextAuthOptions } from 'next-auth';
import GoogleProvider from 'next-auth/providers/google';
import GitHubProvider from 'next-auth/providers/github';
import CredentialsProvider from 'next-auth/providers/credentials';
import { prisma } from './prisma';
import { env } from './env';

const adminEmails = (env.ADMIN_EMAILS ?? '')
  .split(',')
  .map((s) => s.trim().toLowerCase())
  .filter(Boolean);

export const authOptions: NextAuthOptions = {
  providers: [
    GoogleProvider({
      clientId: env.GOOGLE_CLIENT_ID,
      clientSecret: env.GOOGLE_CLIENT_SECRET,
    }),
    GitHubProvider({
      clientId: env.GITHUB_CLIENT_ID,
      clientSecret: env.GITHUB_CLIENT_SECRET,
    }),
    ...(process.env.NODE_ENV === 'test'
      ? [
          CredentialsProvider({
            id: 'e2e-test',
            name: 'E2E Test',
            credentials: { email: { type: 'text' } },
            authorize: async (creds) => {
              if (!creds?.email) return null;
              return { id: 'pending', email: creds.email, name: 'Test User' };
            },
          }),
        ]
      : []),
  ],
  session: {
    strategy: 'jwt',
    maxAge: 60 * 60 * 24 * 7,
    updateAge: 60 * 60 * 24,
  },
  secret: env.NEXTAUTH_SECRET,
  pages: { signIn: '/login' },
  callbacks: {
    async signIn({ user }) {
      if (!user.email) return false;
      const role = adminEmails.includes(user.email.toLowerCase()) ? 'ADMIN' : 'USER';
      await prisma.user.upsert({
        where: { email: user.email },
        update: {
          name: user.name ?? undefined,
          avatar: user.image ?? undefined,
          role,
        },
        create: {
          email: user.email,
          name: user.name,
          avatar: user.image,
          role,
        },
      });
      return true;
    },
    async jwt({ token, user, trigger }) {
      if (user?.email) {
        const dbUser = await prisma.user.findUnique({ where: { email: user.email } });
        if (dbUser) {
          token.sub = dbUser.id;
          token.role = dbUser.role;
          token.email = dbUser.email;
        }
      } else if (trigger === 'update' && token.email) {
        // sliding refresh: re-busca pra pegar role atual
        const dbUser = await prisma.user.findUnique({
          where: { email: token.email as string },
        });
        if (dbUser) token.role = dbUser.role;
      }
      return token;
    },
    async session({ session, token }) {
      if (session.user && token.sub) {
        session.user = {
          ...session.user,
          id: token.sub,
          role: token.role as 'USER' | 'ADMIN',
        };
      }
      return session;
    },
  },
};
```

Tipagem expandida em `apps/web/types/next-auth.d.ts` para `Session.user.id` e `Session.user.role`.

### 6.5 Rate limit `/api/auth/*`

Implementação inline em `middleware.ts` antes do export do middleware do NextAuth — função `rateLimit(ip, limit=5, windowMs=60_000)` com `Map<string, { count, windowStart }>` em escopo de módulo. Em rotas `/api/auth/*`, retorna 429 com `Retry-After`.

Limitação registrada: contador in-memory por instância; com múltiplos containers Railway, limite efetivo é `5 × N`. Backlog: substituir por Upstash Ratelimit. Aceitável para MVP.

### 6.6 `lib/api.ts` — fetch server-side com Bearer

```ts
import { getToken } from 'next-auth/jwt';
import { type NextRequest } from 'next/server';
import { env } from './env';

export async function apiFetch(
  req: NextRequest | Request | undefined,
  path: string,
  init: RequestInit = {},
) {
  // Em RSC, lê o cookie via headers(); em route handlers, via req.
  const token = await getToken({
    req: req as any,
    secret: env.NEXTAUTH_SECRET,
    raw: true,
  });
  return fetch(`${env.API_URL}${path}`, {
    ...init,
    headers: {
      ...(init.headers ?? {}),
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      'Content-Type': 'application/json',
    },
    cache: 'no-store',
  });
}
```

`raw: true` retorna o JWT serializado (não o objeto decodificado) — é o que o Nest precisa receber.

### 7.3 `JwtStrategy`

```ts
@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor(
    config: ConfigService,
    private prisma: PrismaService,
  ) {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: config.getOrThrow<string>('NEXTAUTH_SECRET'),
      algorithms: ['HS256'],
    });
  }

  async validate(payload: { sub: string; email?: string; role?: 'USER' | 'ADMIN' }) {
    if (!payload.sub) throw new UnauthorizedException();
    const user = await this.prisma.user.findUnique({ where: { id: payload.sub } });
    if (!user) throw new UnauthorizedException();
    return user; // vira req.user
  }
}
```

**`role` do payload é ignorado** — sempre se busca o role atual no DB. Isso fecha o gap de invalidação imediata (D8): `updateAge` 24h re-emite o token, mas mesmo dentro da janela um admin despromovido perde o acesso na próxima request. Custo: 1 query/request. Aceitável; cache em backlog.

### 7.4 Guards e decorators

- `JwtAuthGuard extends AuthGuard('jwt')` — registrado **global** em `AppModule` via `APP_GUARD`. Honra `@Public()` reflexão pra liberar rotas (atualmente apenas `/health`; auth roda toda no Next).
- `RolesGuard` registrado também global (depois do `JwtAuthGuard`). Lê `@Roles('ADMIN')` e compara com `req.user.role`. Sem `@Roles`, permite qualquer user autenticado.
- `@CurrentUser()`: `createParamDecorator((_, ctx) => ctx.switchToHttp().getRequest().user)`.
- `@Public()`: `SetMetadata('isPublic', true)`.
- `@Roles(...roles: Role[])`: `SetMetadata('roles', roles)`.

`apps/web/e2e/auth.spec.ts`:

```ts
test('login via credentials provider e fetch /me', async ({ page }) => {
  await page.goto('/login');
  // Em NODE_ENV=test, /login expõe link "E2E Test" oculto.
  // Estratégia: ir direto pra /api/auth/signin/e2e-test via form POST.
  await page.evaluate(() => {
    const form = document.createElement('form');
    form.method = 'POST';
    form.action = '/api/auth/callback/e2e-test';
    const csrf = document.cookie.match(/next-auth\.csrf-token=([^;]+)/)?.[1] ?? '';
    for (const [k, v] of Object.entries({
      csrfToken: decodeURIComponent(csrf).split('|')[0],
      email: 'playwright@test.local',
      callbackUrl: '/',
    })) {
      const input = document.createElement('input');
      input.name = k;
      input.value = v;
      form.appendChild(input);
    }
    document.body.appendChild(form);
    form.submit();
  });
  await page.waitForURL('/');
  await expect(page.getByText('Bem-vindo, Test User')).toBeVisible();

  const me = await page.request.get('/api/v1/me');
  expect(me.ok()).toBe(true);
  const data = await me.json();
  expect(data.email).toBe('playwright@test.local');
  expect(data.role).toBe('USER');
});

test('logout volta pra /login', async ({ page }) => {
  // ...login antes...
  await page.getByRole('button', { name: /menu do usuário/i }).click();
  await page.getByRole('menuitem', { name: 'Sair' }).click();
  await page.waitForURL('/login');
});

test('rota protegida sem login redireciona', async ({ page }) => {

});
```

## 10. Riscos e itens registrados

| #   | Risco                                                                                                          | Mitigação na F1                                                                                                                                                     | Fallback se falhar                                                                                                                                                                                                     |
| --- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R1  | NextAuth v4 pode emitir **JWE** em vez de JWS, dependendo de config — `passport-jwt` HS256 não decodifica JWE. | **Primeira task da implementação:** prova de conceito gerando token de login real, decodificando no Nest com `passport-jwt` HS256. Confirmar `algorithm` e formato. | Se for JWE: substituir `passport-jwt` por `jose.jwtDecrypt` com a `NEXTAUTH_SECRET` derivada via HKDF (NextAuth usa `derive_encryption_key`). Implementar `JwtStrategy` manual sem Passport. Já temos `jose` nas deps. |
| R2  | Rate limit in-memory falha com múltiplas instâncias.                                                           | Anotado em `middleware.ts` com `// TODO(F-backlog)`.                                                                                                                | Backlog: Upstash Ratelimit.                                                                                                                                                                                            |
| R3  | `role` no JWT pode ficar 24h stale.                                                                            | `JwtStrategy.validate` busca role fresca no DB.                                                                                                                     | n/a — comportamento correto.                                                                                                                                                                                           |
| R4  | OAuth callback URLs locais ≠ prod.                                                                             | Documentado em README.                                                                                                                                              | n/a — operacional.                                                                                                                                                                                                     |
| R5  | Privacy/Termos linkados como `#`.                                                                              | Comentário `// TODO(F5)` no componente.                                                                                                                             | F5 implementa páginas estáticas.                                                                                                                                                                                       |
| R6  | E2E depende de `NODE_ENV=test` e Credentials Provider — risco de o provider vazar pra prod por engano.         | `if (process.env.NODE_ENV === 'test')` envolto em test obrigatório no PR template + lint rule (custom ESLint que falha se `e2e-test` aparecer fora desse `if`).     | Test review manual no PR.                                                                                                                                                                                              |
