# F3.1 — AI Runtime + LlmConfig dinâmico (Spec)

> Substrato comum de IA para extração e chat: refatora a chamada direta da OpenAI SDK para um runtime provider-agnóstico (Vercel AI SDK) com modelo + prompt persistidos em DB e gerenciados por admin.

## 1. Goal

Substituir a chamada direta `openai.chat.completions.parse` espalhada no domínio OCR por uma fachada única — `AiRuntime` — que:

1. Resolve `model + prompt + params` em runtime a partir de uma `LlmConfig` ativa por `key` (`EXTRACTOR`, `CHAT`).
2. Suporta múltiplos providers (OpenAI default, Anthropic e Google opcionais via env) através do Vercel AI SDK (`ai` package).
3. Permite ao admin criar novas versões de prompt/modelo, testar contra uma sample, e ativar — tudo via UI, sem deploy.
4. Mantém o contrato de tipos atual (`InvoiceSummaryResult` via Zod) intacto para os consumidores.

A fase **não** mexe em pipeline de classificação (F3.2), validação manual (F3.2) ou histórico de benchmarks (F3.3) — apenas prepara o terreno.

## 2. Contexto e motivação

### Estado atual

- `apps/api/src/ocr/providers/openai-ocr.provider.ts:11` instancia `new OpenAI(...)` direto e usa `chat.completions.parse` com `zodResponseFormat`.
- `apps/api/src/ocr/prompts/extractor.system.ts:1` é um TS exportando string literal — alterar prompt = commit + redeploy.
- `OCR_MODEL` vem de `ConfigService.get('OCR_MODEL') ?? 'gpt-4o'` no construtor — alteração também exige redeploy.
- `BenchmarkModule` usa `OpenAiOcrProvider` direto (`apps/api/src/ocr/benchmark/benchmark.module.ts:8`), o que acopla o benchmark ao provider e ao prompt hard-coded.
- Chat (`apps/api/src/chat/`) ainda não existe; quando vier, herdaria o mesmo problema.

### Por que agora

Três pressões convergem:

1. F3.2 vai precisar adicionar campos novos ao schema (`documentType`, `confidence`) — ergonomia de iterar prompt importa.
2. F3.3 (histórico de benchmark) só faz sentido se cada run aponta para uma `LlmConfig` versionada.
3. Chat (próximo módulo de domínio) deveria nascer já em cima do runtime.

### Divergência consciente vs. CLAUDE.md

CLAUDE.md §"OpenAI único" e §"SDK direto da OpenAI, não LangChain" estabelecem o padrão atual. Esta fase **registra divergência** justificada:

- **Provider**: OpenAI continua sendo o default e o único exigido. Anthropic/Google são habilitáveis por env (chave opcional), nunca obrigatórios. Não é "multi-provider" no sentido do backlog — é "swap-friendly se a chave existir".
- **SDK**: Vercel AI SDK (`ai`) **não é LangChain**. É um wrapper fino sobre as SDKs nativas, sem abstrações pesadas (sem chains, agents, retrievers). A API `generateObject({ model, prompt, schema })` é diretamente análoga ao `chat.completions.parse + zodResponseFormat` que já usamos.

Esta divergência fica registrada aqui e replica-se no header de `apps/api/src/ai-runtime/README.md` quando criado.

## 3. Escopo

### In-scope

- Novo módulo `apps/api/src/ai-runtime/` com facade `AiRuntime`.
- Nova tabela `LlmConfig` (Prisma).
- Refactor `OpenAiOcrProvider` → `ExtractorService` que delega ao `AiRuntime`.
- `MockOcrProvider` mantido como está (continua atendendo `OcrProvider` interface localmente).
- `BenchmarkModule` consome `AiRuntime` em vez de `OpenAiOcrProvider`.
- Admin UI em `/admin/llm-configs`: list, create version, activate, test single sample.
- Seed inicial: `LlmConfig { key: EXTRACTOR, version: 1, model: 'gpt-4o', prompt: <copy de extractor.system.ts>, active: true }` via migration script.

### Out-of-scope (explícito)

- Classificação de documentos / `documentType` / `REJECTED` → F3.2.
- Edição manual de campos extraídos → F3.2.
- Histórico persistido de benchmark runs → F3.3.
- Módulo `chat` (mas `LlmConfigKey.CHAT` já é provisionado no enum).
- Streaming de respostas (Vercel AI SDK suporta nativo, mas não é necessário para extractor — extractor é one-shot).
- Tool calling / function calling (será necessário no chat, F4 ou superior).
- A/B testing automático entre versões de prompt.

## 4. Decisões

| #   | Decisão                                                                                                                           | Alternativa rejeitada                         | Motivação                                                                                           |
| --- | --------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| D1  | Vercel AI SDK (`ai` package) como runtime                                                                                         | OpenAI SDK direto + adapter próprio           | Multi-provider de graça; `generateObject` mapeia 1:1 do uso atual; mantido fino (sem LangChain)     |
| D2  | `LlmConfig` versionado por `key + version` com flag `active`                                                                      | Tabela `current_config` única, sem versão     | F3.3 exige snapshot histórico; rollback fica trivial (toggle `active`)                              |
| D3  | Cache de config ativa em memória, TTL 60s + reload manual via endpoint admin                                                      | Hot-reload por evento (Pub/Sub, NOTIFY)       | Overkill para escopo do case; admin sabe que ativou nova versão e pode forçar reload                |
| D4  | Snapshot do prompt no `BenchmarkRun` (texto literal) — F3.3 referência                                                            | FK pura para `LlmConfig`                      | Mesmo após edição/exclusão da config, histórico continua reproducível                               |
| D5  | Prompt persistido em coluna `Text` (não `JSON`)                                                                                   | JSON com partes (system/user templates)       | Templates por enquanto são strings simples; quando precisar de partials/templating, evolui o schema |
| D6  | `params: Json` (temperature, maxTokens, topP, etc.) — não colunas tipadas                                                         | Colunas dedicadas                             | Diferentes providers expõem params diferentes; JSON é honesto sobre essa heterogeneidade            |
| D7  | Provider whitelist via env: `OPENAI_API_KEY` (default), `ANTHROPIC_API_KEY` (opcional), `GOOGLE_GENERATIVE_AI_API_KEY` (opcional) | Lista no DB                                   | Chaves não devem estar em DB; env é o lugar canônico de credenciais                                 |
| D8  | Modelos disponíveis = whitelist hard-coded em código (`available-models.ts`) filtrada pela env                                    | DB ou descoberta dinâmica via API do provider | Lista é pequena, muda devagar, é compliance-relevant; queremos revisão em PR                        |
| D9  | Endpoint "test" admin executa `AiRuntime` em modo dry-run **sem persistir** — usa um único arquivo do `samples/`                  | Dry-run com input arbitrário do admin         | Mantém superficie pequena; admin valida com sample real do dataset                                  |
| D10 | Validação Zod do payload retornado fica DENTRO do `AiRuntime.generateObject` (não no consumidor)                                  | Cada consumidor valida                        | Garantia de contrato single-source; consumidor recebe `T` já parseado                               |

## 5. Data model

### Schema delta (Prisma)

```prisma
enum LlmConfigKey {
  EXTRACTOR
  CHAT
}

model LlmConfig {
  id        String       @id @default(cuid())
  key       LlmConfigKey
  version   Int
  model     String       // p.ex. "gpt-4o", "claude-sonnet-4-5", "gemini-2.0-flash"
  prompt    String       @db.Text
  params    Json         // { temperature?: number, maxTokens?: number, topP?: number, ... }
  active    Boolean      @default(false)
  notes     String?      @db.Text   // changelog livre escrito pelo admin
  createdBy String
  creator   User         @relation(fields: [createdBy], references: [id], onDelete: Cascade)
  createdAt DateTime     @default(now())

  @@unique([key, version])
  @@index([key, active])
}

model User {
  // ...campos existentes
  llmConfigs LlmConfig[]
}
```

### Invariantes

- Para cada `key`, no máximo **uma** linha com `active = true` em qualquer instante.
- Versões são monotônicas crescentes por `key`: nova versão = `MAX(version) + 1`.
- `LlmConfig` **nunca é deletada** — desativada via `active = false`. (Cascade do `User` é a única exceção; é LGPD.)

### Migration de seed

Migration adicional `20260510_seed_extractor_config.ts` (raw SQL ou script Prisma):

- Cria `LlmConfig { key: EXTRACTOR, version: 1, model: env.OCR_MODEL ?? 'gpt-4o', prompt: <conteúdo de extractor.system.ts>, params: { temperature: 0 }, active: true, createdBy: <primeiro ADMIN ou user system>, notes: 'Seed: prompt original importado de extractor.system.ts' }`.
- Se não houver ADMIN ainda no DB (deploy zerado), o seed cria um `User` system técnico (`email: 'system@invoices-ocr.local'`, `role: ADMIN`) e usa `createdBy: system.id`. Esse user é flagrado como `system` por convenção do email — não vai pelo NextAuth.

### API pública do `AiRuntime`

```ts
// apps/api/src/ai-runtime/ai-runtime.service.ts
@Injectable()
export class AiRuntimeService {
  async generateObject<T extends z.ZodSchema>(opts: {
    key: LlmConfigKey;
    schema: T;
    messages: CoreMessage[]; // tipo do `ai` package
    overrides?: Partial<{ model: string; prompt: string; params: object }>; // só usado pelo "test" endpoint
  }): Promise<z.infer<T>>;
}
```

- Resolve config (cache 60s).
- `system` message = `prompt` da config (sobrescreve qualquer system message dentro de `messages`).
- Combina `params` da config com `temperature: 0` default.
- Chama `generateObject({ model, schema, messages, ...params })` da `ai` SDK.
- Retorna `result.object` — já validado pelo schema.
- Erros: refusal/parse/quota são propagados; não mascara.

### `ProviderRegistry`

```ts
// providers/provider-registry.ts
export function modelFor(modelId: string): LanguageModel {
  if (modelId.startsWith('gpt-')) return openai(modelId);
  if (modelId.startsWith('claude-')) return anthropic(modelId);
  if (modelId.startsWith('gemini-')) return google(modelId);
  throw new BadRequestException(`Unknown model id: ${modelId}`);
}
```

Cada `openai()`/`anthropic()`/`google()` é importado de `@ai-sdk/openai`/`@ai-sdk/anthropic`/`@ai-sdk/google` e lê a env automaticamente. Se a env não estiver setada para um provider, a factory falha em runtime — capturado pela `available-models.ts` antes de oferecer ao admin.

### `available-models.ts`

```ts
const CATALOG = [
  { id: 'gpt-4o', provider: 'openai', requires: 'OPENAI_API_KEY', vision: true },
  { id: 'gpt-4o-mini', provider: 'openai', requires: 'OPENAI_API_KEY', vision: true },
  { id: 'claude-sonnet-4-5', provider: 'anthropic', requires: 'ANTHROPIC_API_KEY', vision: true },
  {
    id: 'gemini-2.0-flash',
    provider: 'google',
    requires: 'GOOGLE_GENERATIVE_AI_API_KEY',
    vision: true,
  },
];

export function availableModels(env: NodeJS.ProcessEnv) {
  return CATALOG.filter((m) => Boolean(env[m.requires]));
}
```

Endpoint admin `GET /api/v1/admin/llm-configs/available-models` retorna esta lista.

## 7. APIs (admin)

Todas em `/api/v1/admin/llm-configs`, protegidas por `@Roles(Role.ADMIN)`. Throttle 10/min.

| Método | Path                            | Body                                                         | Resposta                                                                                                                               |
| ------ | ------------------------------- | ------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| GET    | `/llm-configs`                  | —                                                            | `LlmConfigDto[]` (todas as versões de todas as keys)                                                                                   |
| GET    | `/llm-configs/active/:key`      | —                                                            | `LlmConfigDto` ou 404                                                                                                                  |
| POST   | `/llm-configs`                  | `CreateLlmConfigDto { key, model, prompt, params?, notes? }` | `LlmConfigDto` (cria versão `MAX+1`, `active=false`)                                                                                   |
| POST   | `/llm-configs/:id/activate`     | —                                                            | `LlmConfigDto` (transação: desativa anterior, ativa essa, invalida cache)                                                              |
| POST   | `/llm-configs/:id/test`         | `TestLlmConfigDto { sampleFilename: string }`                | `{ ok: true, result: unknown, durationMs: number }` ou `{ ok: false, error: string, errorClass: 'refusal'\|'parse-error'\|'unknown' }` |
| POST   | `/llm-configs/reload-cache`     | —                                                            | `{ invalidated: number }` (forçar reload sem ativar nova versão)                                                                       |
| GET    | `/llm-configs/available-models` | —                                                            | `AvailableModel[]`                                                                                                                     |

`LlmConfigDto` omite `createdBy` (retorna `createdByEmail` resolvido) e nunca expõe API keys.

DTOs em `packages/shared-types/src/llm-config.ts` para reuso no front.

### Validação

- `prompt`: obrigatório, 1..32_000 chars.
- `model`: obrigatório, deve estar em `availableModels(env)` no momento do create.
- `params`: schema Zod admitindo `temperature` (0..2), `maxTokens` (1..16384), `topP` (0..1) — extras passam (forward-compat).
- `notes`: opcional, ≤ 2000 chars.

## 9. Segurança

- Endpoints `/api/v1/admin/llm-configs/*` → `@Roles(Role.ADMIN)` + `@UseGuards(JwtAuthGuard, RolesGuard)`.
- Rate limit: criar/ativar/test 10/min via `@Throttle`.
- Validação dos DTOs com `class-validator` + Zod no payload retornado.
- Prompt do admin é tratado como **dado** ao ser persistido — não é executado nem interpolado em outros prompts; vai literalmente como `system` message para o provider escolhido.
- API keys de providers nunca leem do DB nem aparecem em logs/respostas.
- Endpoint `test` lê apenas arquivos dentro do diretório `samples/invoice-dataset/` resolvido como path absoluto e validado contra path traversal (`path.resolve` + verificação de prefixo).
- Audit: `createdBy` em cada versão. `activate` não tem audit dedicado nesta fase (backlog: `LlmConfigActivation` table).
- LGPD: User cascade já trata o caso (delete user = delete suas configs). Configs criadas pelo `system` user nunca são deletadas porque o user `system` nunca é removido.

## 10. Migração / rollout

### Ordem de execução (dentro do plano de implementação)

1. Adicionar deps: `ai`, `@ai-sdk/openai`, opcionalmente `@ai-sdk/anthropic` e `@ai-sdk/google`. (`zod` já está.)
2. Migration Prisma: criar `LlmConfig` + enum `LlmConfigKey` + relação no `User`.
3. Migration de seed: insere `EXTRACTOR v1` ativa.
4. Implementar `LlmConfigService` + `AiRuntimeService` + `ProviderRegistry` + `availableModels`.
5. Criar `ExtractorService` que delega ao `AiRuntime`.
6. Substituir consumidores: `OcrService.process` e `BenchmarkService` passam a injetar `ExtractorService` (não mais `OpenAiOcrProvider`).
7. Deletar `OpenAiOcrProvider` e seu spec; manter `MockOcrProvider`.
8. `OcrModule` continua exportando o `OcrProvider` token apenas para o mock (testes); produção usa `ExtractorService` direto.
9. Implementar controller admin + DTOs + tests.
10. Implementar UI admin.
11. Smoke local (mock provider em dev) + smoke staging (OpenAI).

### Backward compat

- `summary`/`extractedText` no `Document` continuam idênticos — schema não muda.
- Cliente do `apps/web` não é afetado (exceto a nova rota admin).
- `OCR_MODEL` env permanece lida APENAS pelo seed da primeira migration. Após isso, fonte de verdade é `LlmConfig.model`. Documentar em `apps/api/.env.example`.

### Rollback

Toggle `active` na versão anterior do `EXTRACTOR` resolve regressão de prompt/modelo. Rollback de código exige reverter migration (perde `LlmConfig`) — só justificável se runtime estiver causando incidente.

## 11. Testes

### Crítico (cobertura obrigatória)

- `LlmConfigService`: cria versão `MAX+1`; `activate` é transacional (única `active=true` por key); cache invalida ao ativar; `findActive(key)` cacheia 60s.
- `AiRuntimeService.generateObject`: monta messages corretas; chama `generateObject` da `ai` SDK com model resolvido; valida via Zod; propaga refusal/parse error sem mascarar; aplica `overrides` no modo test.
- `ProviderRegistry.modelFor`: resolve openai/anthropic/google por prefixo; lança `BadRequestException` para id desconhecido.
- `availableModels`: filtra por env presente (mock `process.env`).
- `ExtractorService.extract`: chama `AiRuntime` com `key=EXTRACTOR` e o schema certo; retorna `InvoiceSummaryResult` válido.
- Controller: ownership/role guards aplicados; throttle configurado; `activate` retorna 404 se id não existe; `test` rejeita filename fora do dir de samples.
- E2E (Playwright): admin loga → cria nova versão de extractor → roda `test` em uma sample → ativa → faz upload de doc real → `summary` chega populado (provider mock no E2E é suficiente; o que importa é o caminho atravessar o `AiRuntime`).

### Não-crítico (sanity rápida apenas)

- Snapshot do payload mock em `MockOcrProvider` (já existe).
- Conformance do schema `invoiceSummarySchema` (já coberto).
