# R1: Upload de Documentos

> Requisito: "Implement a simple webpage using React (we recommend the Next.js framework) that allows users to upload an invoice image. Implement feedback mechanisms on the webpage (e.g., progress indicator, success/error messages)."

---

## Visão Geral

O upload é o primeiro passo do pipeline. O usuário arrasta ou seleciona um arquivo (JPG, PNG ou PDF), vê o progresso em tempo real e recebe feedback imediato sobre sucesso ou erro.

**Arquitetura:**

```
Browser → Next.js API Route → NestJS API → Storage → BullMQ
     ↑______________________________________________↓
                    (feedback via polling)
```

### 1.2 Hook de Upload (`useDocumentUpload`)

```typescript
// apps/web/components/features/document-upload/use-document-upload.ts
export interface OptimisticUpload {
  clientId: string;
  filename: string;
  size: number;
  progress: number; // 0..1
}
```

**Fluxo:**

1. Usuário seleciona arquivo(s)
2. Cria objetos `OptimisticUpload` com `progress: 0`
3. Envia via `XMLHttpRequest` (não `fetch`) para ter acesso ao `onprogress`
4. Atualiza progresso em tempo real via `xhr.upload.onprogress`
5. Em sucesso: dispara evento `UPLOAD_QUEUED_EVENT` para atualizar a lista
6. Em erro: mostra toast com código traduzido

**Por que XMLHttpRequest em vez de fetch?**

- `fetch` não expõe progresso de upload nativamente
- `XMLHttpRequest` tem `xhr.upload.onprogress` com `e.loaded / e.total`

### 1.3 Upload via XHR

```typescript
function uploadOneViaXhr(
  file: File,
  onProgress: (p: number) => void,
): Promise<{ ok: boolean; status: number; body: ApiError | null }> {
  return new Promise((resolve) => {
    const fd = new FormData();
    fd.append('file', file, file.name);
    const xhr = new XMLHttpRequest();
    xhr.open('POST', '/api/upload', true);
    xhr.upload.onprogress = (e) => {
      if (e.lengthComputable && e.total > 0) onProgress(e.loaded / e.total);
    };
    xhr.onload = () => {
      let body: ApiError | null = null;
      try {
        body = JSON.parse(xhr.responseText);
      } catch {
        body = null;
      }
      resolve({ ok: xhr.status >= 200 && xhr.status < 300, status: xhr.status, body });
    };
    xhr.onerror = () => resolve({ ok: false, status: 0, body: null });
    xhr.send(fd);
  });
}
```

## 2. Backend: Recebendo o Upload

### 2.1 Next.js API Route (Proxy)

```typescript
// apps/web/app/api/upload/route.ts
export async function POST(req: NextRequest) {
  const formData = await req.formData();
  const r = await apiUpload('/api/v1/documents', formData, req);
  const body = await r.text();
  return new NextResponse(body, {
    status: r.status,
    headers: { 'Content-Type': 'application/json' },
  });
}
```

O Next.js serve como **BFF** (Backend for Frontend), repassando o FormData para a API NestJS com o cookie de sessão.

### 2.2 NestJS Controller

```typescript
// apps/api/src/documents/documents.controller.ts
@Controller('api/v1/documents')
@SkipThrottle({ benchmark: true })
export class DocumentsController {
  @Post()
  @Throttle({ upload: { ttl: 60_000, limit: 30 } })
  @UseInterceptors(FileInterceptor('file', { limits: { fileSize: TEN_MB } }))
  create(
    @CurrentUser() user: User,
    @UploadedFile() file: Express.Multer.File,
  ): Promise<DocumentSummaryDto> {
    return this.docs.create(user.id, file);
  }
}
```

### 2.3 Validação do Arquivo

```typescript
// apps/api/src/documents/documents.service.ts (trecho)
const detected = await detectFileType(file.buffer);
if (!detected || !ALLOWED_MIME.includes(detected.mime)) {
  throw new BadRequestException({ code: 'upload.invalid_type' });
}
if (file.size > this.maxBytes) {
  throw new PayloadTooLargeException({ code: 'upload.too_large' });
}
```

**Validações aplicadas:**

1. **Magic bytes** via `file-type` (não confia em extensão)
2. **Tamanho máximo** via Multer (`limits.fileSize: 10MB`)
3. **MIME whitelist** (`image/jpeg`, `image/png`, `application/pdf`)
4. **Sanitização de filename** (`sanitizeFilename`) — remove path traversal

### 2.4 Deduplicação Exata (Hash do Conteúdo)

Antes de criar o documento, o sistema calcula o hash SHA-256 do buffer e verifica se o mesmo arquivo já foi enviado:

```typescript
const contentHash = createHash('sha256').update(file.buffer).digest('hex');

const duplicate = await this.findFinalDuplicate(userId, contentHash);
if (duplicate) {
  const duplicateDoc = await this.createDuplicateDocument({
    userId,
    filename: safeName,
    mime: detected.mime,
    size: file.size,
    contentHash,
    original: duplicate,
  });
  return toSummaryDto(duplicateDoc); // ← retorna imediatamente, sem OCR
}
```

**O que acontece com duplicatas:**

- Se o arquivo é idêntico (mesmo hash) e o original está em status final (`READY`, `REJECTED`, `DUPLICATE`):
  - Cria um novo registro `Document` marcado como `DUPLICATE`
  - Aponta `duplicateOfId` para o original
  - **Não reprocessa OCR** — reaproveita o resultado do original
- Se há race condition (dois uploads idênticos simultâneos), o `isContentHashUniqueError` trata a constraint do banco

**Race condition handling:**

```typescript
try {
  created = await this.prisma.document.create({ data: { contentHash, ... } });
} catch (err) {
  if (!this.isContentHashUniqueError(err)) throw err;
  return this.createDuplicateAfterRace({ userId, filename, mime, size, contentHash });
}
```

### 2.5 Armazenamento

```typescript
const storagePath = `${userId}/${created.id}/original.${ext}`;
await this.storage.put(storagePath, file.buffer);
```

- Arquivos são isolados por usuário no storage
- Caminho: `<userId>/<documentId>/original.<ext>`
- Provider: Railway Volume (dev/test) ou Cloudflare R2 (prod)

## 3. Estados do Documento

O documento nasce com status `QUEUED` e evolui:

```
QUEUED → OCR_RUNNING → READY
              ↓
          FAILED / REJECTED / DUPLICATE
```

| Status        | Significado                                |
| ------------- | ------------------------------------------ |
| `QUEUED`      | Upload recebido, aguardando processamento  |
| `OCR_RUNNING` | Job BullMQ está processando                |
| `READY`       | OCR concluído com sucesso                  |
| `FAILED`      | Erro no OCR (retry automático até esgotar) |
| `REJECTED`    | Tipo não suportado ou confiança baixa      |
| `DUPLICATE`   | Documento duplicado detectado              |
