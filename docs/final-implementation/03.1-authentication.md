## 1. Visão Geral

A autenticação é **stateless** baseada em JWT (JSON Web Token) e segue o modelo **OAuth 2.0** com provedores sociais (Google e GitHub). Não há senha — o usuário entra com sua conta Google ou GitHub existente.

**Por que stateless?**

- Não há sessão no servidor (banco, Redis ou memória)
- O token carrega toda a identidade necessária
- Escalável horizontalmente: qualquer instância da API pode validar o token
- Compatível com arquitetura serverless (Next.js App Router)

**Componentes principais:**

| Componente      | Tecnologia                                           | Função                                |
| --------------- | ---------------------------------------------------- | ------------------------------------- |
| Web (OAuth)     | NextAuth v4                                          | Gerencia fluxo OAuth, assina JWT      |
| Web (S2S)       | `internalFetch`                                      | Comunica com API usando token interno |
| API (Validação) | `JwtStrategy` + `jose`                               | Descriptografa e valida JWT           |
| API (Guardas)   | `JwtAuthGuard`, `RolesGuard`, `InternalServiceGuard` | Protege rotas                         |
| API (S2S)       | `InternalUsersController`                            | Recebe sync de usuário do web         |

## 2. Arquitetura Stateless com JWT

### 2.1 O Token JWE

NextAuth usa **JWE** (JSON Web Encryption), não apenas JWS (assinado). Isso significa:

- O payload é **criptografado**, não apenas assinado
- Quem não tem a chave não consegue ler o conteúdo (privacidade)
- A chave é derivada de `NEXTAUTH_SECRET` via HKDF-SHA256

```
┌─────────────────────────────────────────────┐
│  Cookie: next-auth.session-token            │
│  Conteúdo: JWE (criptografado)              │
│  Chave: HKDF(NEXTAUTH_SECRET, "NextAuth...") │
└─────────────────────────────────────────────┘
```

### 2.2 O que o Token Carrega

Após sincronização, o JWT contém:

```json
{
  "sub": "clx123abc...", // CUID do usuário no banco
  "email": "user@email.com",
  "role": "USER", // ou "ADMIN"
  "iat": 1715431200,
  "exp": 1716036000
}
```

**Importante:** o `sub` não é o ID do Google/GitHub. É o CUID gerado pelo Prisma no banco. Isso garante que a identidade do usuário é controlada pela aplicação, não pelo provedor OAuth.

## 3. Fluxo de Login OAuth

```
Usuário clica "Entrar com Google"
         │
         ▼
┌────────────────────┐
│   NextAuth         │
│   (lib/auth.ts)    │
│                    │
│ 1. Redireciona     │
│    para Google     │
└────────────────────┘
         │
         ▼
Google autentica e redireciona de volta
com um "authorization code"
         │
         ▼
┌────────────────────┐
│   NextAuth         │
│                    │
│ 2. Troca code      │
│    por access token│
│ 3. Obtém profile   │
│    (email, nome)   │
└────────────────────┘
         │
         ▼
Callback jwt() é chamado
         │
         ▼
┌────────────────────┐
│   jwtCallback()    │
│   (lib/auth.ts:28) │
│                    │
│ 4. Chama API:      │
│    POST /internal/ │
│    users/sync      │
│    (com S2S token) │
└────────────────────┘
         │
         ▼
API retorna: { id, email, role }
         │
         ▼
Token JWT é atualizado com:
- sub = id do banco (CUID)
- role = role do banco
         │
         ▼
Cookie é setado no browser
```

### 3.1 Código do `jwtCallback`

```typescript
// apps/web/lib/auth.ts
export async function jwtCallback({ token, user, trigger }: JwtArgs) {
  const shouldSync = (user != null && user.email != null) || trigger === 'update';
  if (!shouldSync) return token;

  const email = user?.email ?? (token.email as string | undefined);
  if (!email) return token;

  // Sincroniza com a API (cria/atualiza usuário no banco)
  const res = await internalFetch('/api/v1/internal/users/sync', {
    method: 'POST',
    body: JSON.stringify({ email, name: user?.name, avatar: user?.image }),
  });

  // Falha aqui é FATAL: se não substituirmos token.sub pelo CUID do User,
  // o NextAuth assina o JWT com o sub do provedor OAuth (Google/GitHub) e
  // a API rejeita TODAS as requests com 401.
  if (!res.ok) {
    throw new Error(`User sync failed (${res.status})`);
  }

  const synced = (await res.json()) as { id: string; email: string; role: Role };
  token.sub = synced.id; // ← CUID do banco, não ID do Google!
  token.role = synced.role;
  token.email = synced.email;

  return token;
}
```

**Design decision:** o `jwtCallback` é síncrono com o login. Se a API estiver offline, o login falha. Isso é intencional — é melhor falhar cedo do que entregar um token inválido que vai ser rejeitado em toda requisição posterior.

## 4. Sincronização de Usuário (Web → API)

### 4.1 Por que existe um sync?

O NextAuth roda no frontend (Next.js). A API (NestJS) é um serviço separado. O banco de dados fica na API. Logo, quando um usuário loga pela primeira vez:

1. NextAuth descobre o email do Google/GitHub
2. Precisa criar (ou atualizar) o registro no PostgreSQL
3. Faz isso chamando a API via S2S (Service-to-Service)

### 4.2 O Endpoint Interno

```typescript
// apps/api/src/users/internal-users.controller.ts
@Controller('api/v1/internal/users')
@SkipThrottle({ benchmark: true })
export class InternalUsersController {
  @InternalOnly() // ← só aceita S2S token
  @Post('sync')
  async sync(@Body() dto: SyncUserDto) {
    const user = await this.users.upsertByEmail({
      email: dto.email,
      name: dto.name,
      avatar: dto.avatar,
    });
    return {
      id: user.id,
      email: user.email,
      name: user.name,
      avatar: user.avatar,
      role: user.role,
    };
  }
}
```

### 4.3 O que é `@InternalOnly()`?

É um decorator que aplica o `InternalServiceGuard`:

```typescript
// apps/api/src/auth/decorators/internal-only.decorator.ts
export const InternalOnly = (): MethodDecorator & ClassDecorator =>
  SetMetadata(INTERNAL_ONLY_KEY, true);
```

O guard verifica o header `x-internal-token`:

```typescript
// apps/api/src/auth/guards/internal-service.guard.ts
export class InternalServiceGuard implements CanActivate {
  canActivate(ctx: ExecutionContext): boolean {
    const raw = req.headers['x-internal-token'];

    // Comparação segura contra timing attacks
    const a = createHash('sha256').update(raw).digest();
    const b = createHash('sha256').update(expected).digest();
    if (!timingSafeEqual(a, b)) throw new UnauthorizedException();
    return true;
  }
}
```

**Segurança:**

- `timingSafeEqual` previne timing attacks (não vaza o tamanho do segredo)
- Hash SHA-256 garante buffers de tamanho fixo (32 bytes) para comparação
- Header só é aceito em endpoints marcados com `@InternalOnly()`

### 4.4 O Cliente S2S no Web

```typescript
// apps/web/lib/internal-api.ts
export function internalFetch(path: string, init?: RequestInit): Promise<Response> {
  return fetch(`${env.API_URL}${path}`, {
    ...init,
    headers: {
      'Content-Type': 'application/json',
      ...(init?.headers as Record<string, string>),
      'x-internal-token': env.INTERNAL_SERVICE_TOKEN, // ← S2S token
    },
  });
}
```

**Restrição:** `internalFetch` só é usado server-side (no `jwtCallback` e SSR). Nunca é exposto ao browser.

## 5. Validação do Token na API

### 5.1 JwtStrategy: Descriptografando o JWE

```typescript
// apps/api/src/auth/strategies/jwt.strategy.ts
export class JwtStrategy {
  private readonly keyPromise: Promise<Uint8Array>;

  constructor(
    config: ConfigService,
    private readonly prisma: PrismaService,
  ) {
    // Deriva a mesma chave que o NextAuth usa para criptografar
    this.keyPromise = deriveEncryptionKey(config.getOrThrow<string>('NEXTAUTH_SECRET'));
  }

  async decryptAndValidate(token: string): Promise<User> {
    let payload: JWTPayload;
    try {
      const key = await this.keyPromise;
      // jwtDecrypt vem da lib 'jose' — descriptografa JWE
      ({ payload } = await jwtDecrypt(token, key, { clockTolerance: 15 }));
    } catch (err) {
      // Loga a mensagem real do jose para diagnóstico:
      // - "decryption operation failed" → secret diferente
      // - "exp claim timestamp check failed" → token expirado
      // - "Invalid Compact JWE" → corrupção em trânsito
      this.logger.warn(`JWE decrypt failed: ${err.message}`);
      throw new UnauthorizedException('invalid token');
    }
    return this.validate(payload as AuthPayload);
  }

  async validate(payload: AuthPayload): Promise<User> {
    if (!payload?.sub) throw new UnauthorizedException();
    const user = await this.prisma.user.findUnique({
      where: { id: payload.sub },
    });
    if (!user) throw new UnauthorizedException();
    return user;
  }
}
```

**Importante:** a chave é derivada do mesmo `NEXTAUTH_SECRET` via HKDF. Isso garante que web e API compartilham a mesma chave de criptografia sem precisar trocar chaves.

### 5.2 JwtAuthGuard: Protegendo Todas as Rotas

```typescript
// apps/api/src/auth/guards/jwt-auth.guard.ts
export class JwtAuthGuard implements CanActivate {
  async canActivate(ctx: ExecutionContext): Promise<boolean> {
    // Rotas marcadas com @Public() não precisam de auth
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      ctx.getHandler(),
      ctx.getClass(),
    ]);
    if (isPublic) return true;

    const req = ctx.switchToHttp().getRequest();
    const auth = req.headers['authorization'];
    const token = typeof auth === 'string' ? auth.replace(/^Bearer\s+/i, '') : '';
    if (!token) throw new UnauthorizedException();

    // Descriptografa o token e anexa o usuário no request
    req.user = await this.strategy.decryptAndValidate(token);
    return true;
  }
}
```

**Configuração global:** o `JwtAuthGuard` é aplicado a **todas** as rotas via `APP_GUARD`:

```typescript
// apps/api/src/app.module.ts
providers: [
  { provide: APP_GUARD, useClass: JwtAuthGuard }, // ← todas as rotas
  { provide: APP_GUARD, useClass: UserScopedThrottlerGuard },
  { provide: APP_GUARD, useClass: RolesGuard },
];
```

**Exceção:** rotas marcadas com `@Public()` ou `@InternalOnly()` são liberadas.

### 5.3 Como o Token Chega na API

```
Browser faz request para API
    │
    ├── Cookie: next-auth.session-token=eyJ... (JWE)
    │
    ▼
Next.js API Route (server-side)
    │
    ├── getServerSession() → descriptografa cookie
    ├── Extrai token do session
    ├── Envia header: Authorization: Bearer <token>
    │
    ▼
NestJS API
    │
    ├── JwtAuthGuard extrai header
    ├── JwtStrategy.decryptAndValidate(token)
    ├── Anexa req.user = { id, email, name, role }
    │
    ▼
Controller pode usar @CurrentUser()
```

## 6. S2S: Comunicação Segura entre Serviços

### 6.1 Por que S2S é necessário?

O web e a API são serviços separados. O web precisa:

- Sincronizar usuário no login (`jwtCallback`)
- Fazer SSR de páginas protegidas (buscar documentos no banco)
- Deletar usuários em testes

Tudo isso requer acesso a endpoints que não devem ser públicos.

### 6.2 Arquitetura S2S

```
┌──────────────┐                    ┌──────────────┐
│   Next.js    │ ──x-internal-token──▶ │   NestJS     │
│   (Web)      │   (SHA-256 compare)   │   (API)      │
│              │ ◀─── 200 / 401 ────── │              │
└──────────────┘                    └──────────────┘
       │                                    │
       │ env.INTERNAL_SERVICE_TOKEN         │ env.INTERNAL_SERVICE_TOKEN
       │ (mesmo valor em ambos)             │ (mesmo valor em ambos)
```

### 6.3 Proteção contra Timing Attacks

```typescript
// apps/api/src/auth/guards/internal-service.guard.ts
const a = createHash('sha256').update(raw).digest();
const b = createHash('sha256').update(expected).digest();
if (!timingSafeEqual(a, b)) throw new UnauthorizedException();
```

Sem o hash SHA-256, `timingSafeEqual` lançaria `RangeError` quando os tamanhos diferem — vazando o tamanho real do segredo. Com hash, ambos têm 32 bytes sempre.

## 7. Middleware de Proteção de Rotas

### 7.1 Next.js Middleware

```typescript
// apps/web/middleware.ts
export default function middleware(req: NextRequest) {
  const { pathname } = req.nextUrl;

  // 1. Rate-limit em endpoints sensíveis de auth
  if (pathname.startsWith('/api/auth/')) {
    if (rateLimitEnabled() && isRateLimited(pathname)) {
      const ip = req.headers.get('x-forwarded-for')?.split(',')[0].trim() || 'unknown';
      const result = rateLimit(ip, SENSITIVE_AUTH_LIMIT); // 10 req
      if (result.limited) {
        return new NextResponse('Too Many Requests', { status: 429 });
      }
    }
    return NextResponse.next();
  }

  // 2. Outras rotas /api/* (proxy/route handlers) decidem auth sozinhas
  if (pathname.startsWith('/api/')) {
    return NextResponse.next();
  }

  // 3. Todas as outras rotas exigem autenticação (withAuth)
  return authMiddleware(req);
}

export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon.ico|login).*)'],
};
```

### 7.2 O que o Middleware Protege

- **Rotas protegidas:** `/documents`, `/chat`, `/admin/*`
- **Rate-limit:** `/api/auth/signin/*` e `/api/auth/callback/*` (10 req)
- **Bypass em dev:** rate-limit não aplica em `NODE_ENV !== 'production'`
- **Bypass em build:** `_next/static`, `_next/image`, `favicon.ico`, `/login`

## 8. RBAC: Controle de Acesso por Papéis

### 8.1 Modelo de Papéis

```prisma
// schema.prisma
enum Role {
  USER
  ADMIN
}
```

### 8.2 Decorator `@Roles()`

```typescript
// apps/api/src/auth/decorators/roles.decorator.ts
export const Roles = (...roles: Role[]) => SetMetadata(ROLES_KEY, roles);
```

### 8.3 RolesGuard

```typescript
// apps/api/src/auth/guards/roles.guard.ts
export class RolesGuard implements CanActivate {
  canActivate(ctx: ExecutionContext): boolean {
    const required = this.reflector.getAllAndOverride<Role[] | undefined>(ROLES_KEY, [
      ctx.getHandler(),
      ctx.getClass(),
    ]);
    if (!required || required.length === 0) return true; // ← sem @Roles() = público

    const user = ctx.switchToHttp().getRequest<{ user?: { role: Role } }>().user;
    if (!user || !required.includes(user.role)) {
      throw new ForbiddenException();
    }
    return true;
  }
}
```

### 8.4 Uso

```typescript
// apps/api/src/admin/metrics/metrics.controller.ts
@Controller('api/v1/admin')
@Roles(Role.ADMIN)           // ← só ADMIN pode acessar
export class MetricsController {
  @Get('metrics')
  getMetrics() { ... }
}
```

## 9. Rate Limiting em Autenticação

### 9.1 Throttler do NestJS

```typescript
// apps/api/src/app.module.ts
ThrottlerModule.forRoot([
  { name: 'default', ttl: 60_000, limit: 600 },
  { name: 'upload', ttl: 60_000, limit: 120 },
  { name: 'ocr', ttl: 60_000, limit: 60 },
  { name: 'benchmark', ttl: 3_600_000, limit: 20 },
  { name: 'chat', ttl: 60_000, limit: 60 },
  { name: 'download', ttl: 60_000, limit: 60 },
]),
```

### 9.2 User-Scoped Throttler

O `UserScopedThrottlerGuard` usa o `userId` como chave do rate-limit (não IP), evitando que um usuário legítimo seja afetado por outro na mesma rede.

### 9.3 Middleware Rate-Limit (Next.js)

```typescript
// apps/web/middleware.ts
function isRateLimited(pathname: string): boolean {
  return pathname.startsWith('/api/auth/signin/') || pathname.startsWith('/api/auth/callback/');
}
```

**Por que só signin/callback?**

- `signin/*` — início do fluxo OAuth (brute-force de provedores)
- `callback/*` — troca de code por token (replay attacks)
- **NÃO** limita: `csrf`, `session`, `signout`, `providers` — são chamadas rotineiras do cliente

## 10. Segurança: Secret Fingerprint

### 10.1 O Problema

`NEXTAUTH_SECRET` é compartilhado entre web e API. Se um serviço tiver um valor diferente (erro de deploy, variável não setada, typo), a descriptografia falha silenciosamente com 401.

### 10.2 A Solução

```typescript
// apps/web/lib/secret-fingerprint.ts
// apps/api/src/auth/util/secret-fingerprint.ts (cópia idêntica)
export function secretFingerprint(secret: string): string {
  return createHmac('sha256', secret).update('fingerprint').digest('hex').slice(0, 8);
}
```

### 10.3 Como Funciona

```typescript
// apps/web/lib/auth.ts
if (process.env.NEXT_PHASE !== 'phase-production-build') {
  console.log(`[bootstrap] NEXTAUTH_SECRET fingerprint: ${secretFingerprint(env.NEXTAUTH_SECRET)}`);
}
```

**Output no log:**

```
[bootstrap] NEXTAUTH_SECRET fingerprint: a3f7b2c1
```

**Diagnóstico:**

- Se web e API imprimem o **mesmo** fingerprint → secret sincronizado ✅
- Se fingerprints **diferem** → mismatch ❌ (independente do dashboard do Railway)

**Bug real:** durante o desenvolvimento, o web e a API tinham secrets diferentes. O fingerprint revelou o problema imediatamente.

## 11. Testes E2E: Provider de Credenciais

### 11.1 O Problema

Testes E2E com Playwright precisam logar automaticamente. Não é viável interagir com popup do Google/GitHub.

### 11.2 A Solução

```typescript
// apps/web/lib/auth.ts
...(process.env.NODE_ENV === 'test' || process.env.E2E_TEST === '1'
  ? [
      CredentialsProvider({
        id: 'e2e-test',
        name: 'E2E Test',
        credentials: { email: { type: 'text' } },
        authorize: async (creds) => {
          if (!creds?.email) return null;
          return { id: 'pending', email: creds.email, name: 'Test User' };
        },
      }),
    ]
  : []),
```

**Uso no teste:**

```typescript
// apps/web/e2e/auth.spec.ts
await page.goto('/api/auth/signin/e2e-test?callbackUrl=/documents');
await page.fill('input[name="email"]', 'test@example.com');
await page.click('button[type="submit"]');
```

**Segurança:** o provider só existe quando `E2E_TEST=1` ou `NODE_ENV=test`. Nunca em produção.

## Resumo do Requisito

O case pedia: "Users must be authenticated prior to being able to upload the invoice."

**Como foi resolvido:**

| Aspecto     | Implementação                              |
| ----------- | ------------------------------------------ |
| Provedores  | Google + GitHub OAuth                      |
| Sessão      | JWT criptografado (JWE) no cookie          |
| Stateless   | ✅ Sem sessão no servidor                  |
| Escalável   | ✅ Qualquer instância valida o token       |
| S2S seguro  | `x-internal-token` com timing-safe compare |
| RBAC        | `USER` / `ADMIN` com `@Roles()`            |
| Rate-limit  | `@nestjs/throttler` + middleware Next.js   |
| Diagnóstico | Secret fingerprint para sincronização      |
| Testes E2E  | Provider de credenciais condicional        |
